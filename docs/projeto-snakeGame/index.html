O que vamos construir?

Nas últimas aulas aprendemos variáveis, condicionais, laços, funções e arrays. Cada coisa isolada, em exemplos pequenos.

Hoje tudo isso se junta em um projeto real. E a estrela da aula é uma velha conhecida da programação: O jogo da cobrinha!

Note: A cobra é um Array

Mais precisamente, um array de objetos, onde cada item guarda a posição x e y de um quadradinho do corpo.

// A cobra, no começo do jogo:
snake = [
    { x: 270, y: 270 }  // só a cabeça
    // essa posição é dada em pixels
    // e a definimos por meio de boxsize * 9
    // boxsize = 30, daí 270 para x e y 
    // Como a grid é 600x600, seu centro 
    // está no quadrado que começa com o canto
    // superior esquerdo em (270, 270)
];

Todo o "movimento" que você vê na tela é o resultado de mexer nesse array várias vezes por segundo.

HTML5: a tag <canvas>

O <canvas> é literalmente uma tela de pintura em branco. Diferente de uma <div>, ele não tem filhos, não tem texto, não tem layout: ele só tem pixels.

Quem desenha nele é o JavaScript, através de um objeto chamado contexto:

  • O texto dentro da tag só aparece se o navegador for antigo demais, chamamos isso de fallback.
  • getContext("2d") devolve a "caixa de pincéis" que usaremos o jogo inteiro.
<canvas class="gameCanva" 
        width="600" 
        height="600">
    O jogo da "cobrinha" não pode
    ser exibido no seu navegador
</canvas>

const canvas = document.querySelector(".gameCanva");
const ctx = canvas.getContext("2d");
// ctx = nosso "pincel"

Atributo vs. CSS

Esse é o erro nº 1 de quem começa com canvas. Preste muita atenção:

Atributo HTML

<canvas width="600" height="600">

Define quantos pixels de verdade existem dentro da tela. É o tamanho da folha de papel.

Propriedade CSS

canvas { width: 600px; }

Define o tamanho que a folha aparenta ter na tela. É o zoom aplicado na folha depois de pronta.

Analogia da fotografia

Imagine uma foto 3x4. O atributo é a resolução da foto (a quantidade de pixels que existe nela). O CSS é o tamanho do porta-retratos onde você a coloca.

Se a foto tem 100 pixels e você a coloca num porta-retrato de 600px, ela estica e borra. É exatamente isso que acontece se você definir o tamanho do canvas só pelo CSS: os desenhos ficam esticados e as contas do jogo param de bater (o mouse clica num lugar e o jogo acha que foi em outro).

Por isso, no nosso jogo, o tamanho está no atributo da tag, e o CSS não mexe no tamanho do canvas.

CSS3: Posicionamento e Camadas

Nosso menu ("Cobrinha! / Jogar") precisa flutuar por cima do canvas, bem no centro. Sem Flexbox (que veremos no final do curso), resolvemos isso com posicionamento:

.gameContainer {
    position: relative;
    overflow: hidden;
}

.menuGame {
    position: absolute;
    top: 50%;
    left: 50%;
    transform: translate(-50%, -50%);
    z-index: 10;
}

canvas {
    display: none;
}
  • position: relative no pai — ele vira o "mapa de referência". Sozinho não muda nada, mas avisa: "filhos absolutos, contem as coordenadas a partir de mim".
  • position: absolute no menu — ele sai do fluxo normal da página (deixa de "empurrar" os vizinhos) e flutua livremente dentro do container.
  • top/left: 50% + translate(-50%,-50%) — o 50% joga o canto superior esquerdo do menu no centro; o translate puxa o menu de volta por metade do próprio tamanho. Resultado: centralizado de verdade.
  • z-index: 10 — controla a altura das camadas (eixo Z, saindo da tela em direção aos seus olhos). Quem tem número maior fica na frente.
  • overflow: hidden — tudo que passar da borda do container é cortado. Junto com o border-radius, ele garante os cantos arredondados.

Por que display: none no canvas?

Quando a página carrega, quem deve aparecer é o menu, não o jogo. Então o canvas nasce escondido:

canvas {
    display: none; /* O canvas NÃO RENDERIZA */
}

E o JavaScript faz a troca de telas na hora certa, dentro da função initGame():

function initGame(){
    startMenu.style.display = "none";  // some o menu
    canvas.style.display = "block";    // aparece o jogo
    ...
}

Repare que isso é um padrão que você vai usar a vida inteira no front-end: em vez de criar e destruir elementos, você só liga e desliga a visibilidade deles.

display: none vs. visibility: hidden

display: none remove o elemento do layout (é como se ele não existisse). visibility: hidden apenas o torna invisível, mas o buraco dele continua ocupando espaço na página.

No fim do jogo (game over), fazemos o caminho inverso: escondemos o canvas e mostramos o menu de novo.

A Malha (o Grid invisível)

O canvas tem 600 pixels. Mas a cobra não anda de 1 em 1 pixel — ela pula de quadradinho em quadradinho. Nós dividimos a tela numa matriz:

const boxSize = 30;      // lado do quadrado
const canvasSize = 600;  // lado do canvas

// Quantos quadrados cabem por linha?
// 600 / 30 = 20

// Total de casas do tabuleiro:
// 20 x 20 = 400 casas

Como 600 é divisível por 30, a divisão é exata: não sobra "meio quadrado" na borda. Se o boxSize fosse 40, teríamos 15 casas — também funcionaria. Se fosse 32, sobrariam 24 pixels quebrados e a cobra nunca encostaria perfeitamente na parede.

Todo elemento do jogo (cada pedaço da cobra e a maçã) ocupa exatamente uma casa de 30x30. Por isso tudo se encaixa.

A malha 20x20 (com a cobra e a maçã ocupando casas inteiras):

Coordenadas: a casa da coluna 9, linha 9 vale x = 9 * 30 = 270 e y = 270. É exatamente onde a cabeça nasce.

O Loop do Jogo

Um jogo não é uma "animação". É um desenho refeito do zero, várias vezes por segundo. Como um flipbook (aqueles desenhos animados feitos nas quinas do caderno).

// setInterval(função, intervalo_em_ms)
// A cada 130 milissegundos, chame drawGame() de novo.
gameInterval = setInterval(drawGame, 130);

E a primeira coisa que drawGame() faz é apagar o frame anterior, pintando um retângulo verde do tamanho da tela inteira por cima de tudo:

function drawGame() {
    // 1. Apaga tudo
    ctx.fillStyle = "#e8f5e9";
    ctx.fillRect(0, 0, canvasSize, canvasSize);

    // 2. Redesenha a cobra
    // 3. Redesenha a maçã
    // 4. Move o array
    ...
}
  • ctx.fillStyle — escolhe a cor da tinta do preenchimento. Sozinho ele não desenha nada! É como pegar o pincel e molhar na cor.
  • ctx.fillRect(x, y, largura, altura) — desenha um retângulo preenchido. O x, y é o canto superior esquerdo.
  • ctx.strokeStyle — cor do contorno (a caneta).
  • ctx.strokeRect(x, y, l, a) — desenha só o contorno, sem preencher.

No nosso jogo, o strokeRect é desenhado na cor do fundo, criando aquela linha de separação entre os quadradinhos do corpo da cobra. Por padrão ocupa 1px

Se você esquecer de apagar: a cobra vira um rastro sólido crescendo pela tela, porque cada frame novo é desenhado por cima do antigo, e o antigo nunca some. Teste isso! Comente as duas primeiras linhas do drawGame e veja o que acontece.

A Maçã: sorteio de posição

A maçã precisa nascer numa casa válida da malha — nunca "entre" dois quadrados. Veja a receita:

function drawApple(){
    apple = {
        x: Math.floor(
            Math.random() * (canvasSize / boxSize)
        ) * boxSize,

        y: Math.floor(
            Math.random() * (canvasSize / boxSize)
        ) * boxSize
    }
}

Vamos ler de dentro para fora, um passo por vez:

  • Math.random() → sorteia um decimal entre 0 e 0.999… Ex: 0.734
  • * (600 / 30) → multiplica por 20 → 14.68
  • Math.floor(...) → arredonda para baixo → 14 (agora é o índice de uma coluna válida: 0 a 19)
  • * boxSize → converte índice em pixel → 14 * 30 = 420

Resultado: a maçã sempre cai exatamente em cima de uma casa da malha, no mesmo "sistema de coordenadas" da cobra. Isso é o que torna a colisão possível!

Geometria: o círculo dentro do quadrado

ctx.fillStyle = "#e53935";
ctx.beginPath();
ctx.arc(
    apple.x + boxSize / 2,  // centro X
    apple.y + boxSize / 2,  // centro Y
    boxSize / 2.2,          // raio
    0,                      // ângulo inicial
    Math.PI * 2             // ângulo final (360°)
);
ctx.fill();

Atenção à diferença: no fillRect, o x, y é o canto superior esquerdo. No ctx.arc, o x, y é o centro do círculo!

Por isso somamos boxSize / 2 (metade da casa) nas duas coordenadas: assim o centro do círculo cai no centro do quadrado.

beginPath() = "começar um desenho novo" (apaga o traçado anterior da memória, senão os círculos antigos seriam redesenhados junto, criando um rastro). fill() = "agora sim, pinte esse traçado". Só isso — voltemos ao JS de verdade.

Para o jogador, a maçã é um círculo.
Para a engine, ela é um quadrado de 30x30.

↕ folga (raio = 30/2.2 ≈ 13.6px)

O raio é boxSize / 2.2 e não boxSize / 2 apenas por estética: com /2 o círculo encostaria nas quatro bordas e ficaria colado nos vizinhos. O 2.2 deixa uma folguinha, e a maçã "respira" dentro da casa.

Mas atenção: a colisão NÃO usa o círculo. Ela compara x e y da casa. Por isso a cobra parece comer a maçã "sem encostar" nela — ela encostou na hitbox, o retângulo invisível.

JavaScript é Orientado a Eventos

Até agora nossos programas eram uma fila: linha 1, linha 2, linha 3, fim. O JavaScript do navegador não funciona assim.

Ele fica parado, esperando algo acontecer. Um clique, uma tecla, o carregamento da página. Quando acontece, ele dispara a função que estava "de plantão".

Isso é um Event Listener — o "secretário de eventos" que anota o que aconteceu e avisa quem precisa saber.

// alvo.addEventListener(evento, função)

startBtn.addEventListener("click", initGame);
//   ^quem      ^o que      ^chame quem

Repare: passamos initGame sem os parênteses. Estamos entregando a função em si ("guarde esta receita"). Com parênteses, ela executaria na hora — e o jogo começaria sozinho!

Escutando o teclado

document.addEventListener("keydown", (event) => {
    let key = event.keyCode;
    const arrayKeys = [37, 38, 39, 40];

    if (arrayKeys.includes(key)) {
        event.preventDefault();
    }

    if (key === 38 && direction != "DOWN") {
        direction = "UP";
    } else if (key === 37 && direction != "RIGHT") {
        direction = "LEFT";
    } else if (key === 39 && direction != "LEFT") {
        direction = "RIGHT";
    } else if (key === 40 && direction != "UP") {
        direction = "DOWN";
    }
});
  • keydown = a tecla foi pressionada.
  • 37, 38, 39, 40 = esquerda, cima, direita, baixo. Reconheceu o includes()? É o mesmo da aula passada, agora num caso real!
  • preventDefault() impede o comportamento padrão do navegador (as setas rolariam a página).
  • A condição && direction != "DOWN" impede a cobra de dar meia-volta em si mesma e morrer instantaneamente.
  • O evento só muda uma variável (direction). Quem move a cobra continua sendo o loop.

O Movimento: uma ilusão de Array

Aqui está o coração da aula. A cobra não "anda". A cada frame, o JavaScript faz apenas duas operações no array:

// 1. Onde está a cabeça agora?
let snakeX = snake[0].x;
let snakeY = snake[0].y;

// 2. Para onde ela vai?
if (direction == "LEFT")  snakeX -= boxSize;
if (direction == "UP")    snakeY -= boxSize;
if (direction == "RIGHT") snakeX += boxSize;
if (direction == "DOWN")  snakeY += boxSize;

// 3. Remove a cauda (o último item)
snake.pop();

// 4. Insere a nova cabeça no ÍNDICE 0
let newHead = { x: snakeX, y: snakeY };
snake.unshift(newHead);

Pense numa fila de pessoas

Todo mundo quer "andar para frente". Mas ninguém precisa se mexer! Basta que:

  • Uma pessoa nova entre na frente da fila → unshift()
  • A última pessoa saia da fila → pop()

De longe, parece que a fila inteira andou um passo. Mas os do meio ficaram parados. Essa é exatamente a ilusão da cobrinha.

E quando ela come? Simplesmente não fazemos o pop(). A cabeça nova entra, ninguém sai, e o array cresce em 1. Pronto: a cobra ficou mais comprida.

Relembrando os métodos: push() adiciona no fim, pop() remove do fim, unshift() adiciona no início (índice 0) e shift() remove do início. Aqui usamos a dupla unshift + pop: entra na frente, sai por trás.

Sistema de Colisões

Colisão, aqui, é só comparação de números. Nada de física complicada. São três checagens:

1. Colisão com as PAREDES

if (snakeX < 0 || snakeX >= canvasSize || snakeY < 0 || snakeY >= canvasSize)

A cabeça saiu pela esquerda (x < 0), pela direita (x >= 600), por cima (y < 0) ou por baixo (y >= 600)? Repare no >=: a última casa válida começa em 570 (19 × 30). Se o x virasse 600, a cobra já estaria com o corpo fora da tela.

2. Colisão com o PRÓPRIO CORPO

function checkCollision(headX, headY, array) {
    for (let i = 0; i < array.length; i++) {
        if (headX == array[i].x && headY == array[i].y) {
            return true;
        }
    }
    return false;
}

Aqui está o laço for percorrendo o array exatamente como treinamos na aula passada, com .length! A função varre cada pedaço do corpo e pergunta: "a nova cabeça vai cair em cima de você?". Se alguém responder que sim, ela retorna true e o jogo acaba.

3. Colisão com a MAÇÃ

if (snakeX === apple.x && snakeY === apple.y) {
    drawApple();   // sorteia uma maçã nova... e NÃO faz pop()
} else {
    snake.pop();   // não comeu: a cauda vai embora
}

O if/else aqui decide entre crescer e andar. É por isso que o array e o corpo da cobra sempre têm o mesmo tamanho.

Fim de jogo: chamamos clearInterval(gameInterval) para desligar o loop (senão o jogo continuaria desenhando para sempre), mostramos o alert com o placar e trocamos as telas de volta.

Resumo da Aula

O que aprendemos hoje

  • <canvas>: uma tela de pixels controlada 100% pelo JS.
  • Atributo ≠ CSS: resolução da folha vs. zoom da folha.
  • position + z-index: empilhando o menu por cima do jogo.
  • Malha: 600 / 30 = 20 casas. Tudo se alinha porque a divisão é exata.
  • Loop: apagar tudo e redesenhar, várias vezes por segundo.
  • Eventos: o JS espera, o usuário age, a função responde.
  • Colisão: comparar x e y.

E a grande sacada

Um jogo inteiro, que parece complexo, se resume a:

snake.unshift(novaCabeca);  // entra na frente
snake.pop();                // sai por trás

Aqueles métodos de array que pareciam abstratos na aula passada — push, pop, includes, length — são exatamente as ferramentas que fazem um jogo funcionar.

Arrays guardam o estado. Funções organizam as regras. O loop dá vida.

É assim em todo jogo: do Snake ao seu Battle Royale favorito. Muda a escala, não a ideia.
